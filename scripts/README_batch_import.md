# 批量日志导入脚本

这个脚本用于将时间跟踪数据从 `.txt` 文件批量导入到 Common Compass 系统中。

## 功能特性

- 📁 **文件解析**: 支持解析制表符分隔的 `.txt` 文件
- 🔄 **数据转换**: 自动将原始数据转换为 API 格式
- 📊 **维度映射**: 自动映射生命维度到对应的数据库 ID
- 🛡️ **数据验证**: 使用 Pydantic 进行严格的数据验证
- 📝 **日志记录**: 详细的处理日志，便于调试
- 🔍 **预览模式**: 支持 dry-run 模式，预览处理结果
- 🔌 **可复用客户端**: 提供可复用的 API 客户端函数
- 📄 **JSON 输出**: 可选的 JSON 格式输出，便于数据审查

## 输入文件格式

输入文件应为制表符分隔的文本文件，包含以下 6 列：


| 字段 | 格式 | 说明 | 示例 |
|------|------|------|------|
| 日期 | YYYY-MM-DD | 活动发生的日期 | 2025-08-08 |
| 开始时间 | HH:MM | 活动开始时间（24小时制） | 12:20 |
| 结束时间 | HH:MM | 活动结束时间（24小时制），支持跨天 | 0:00 |
| 分钟数 | 整数 | 活动持续时间（分钟） | 5 |
| 维度 | 字符串 | 生命维度分类 | 5-财富 |
| 活动描述 | 字符串 | 活动的详细描述 | 记录日志 |

### 跨天时间段处理

脚本自动处理跨天的时间段：

- **同一天内**：`2025-07-30	22:35	22:45` → `2025-07-30T22:35:00` 到 `2025-07-30T22:45:00`
- **跨天时间段**：`2025-07-30	23:20	0:00` → `2025-07-30T23:20:00` 到 `2025-07-31T00:00:00`
- **多天数据**：支持连续多天的数据，每个时间段都会正确映射到对应的日期


## 自动维度处理

脚本具备自动维度处理功能，无需手动初始化：

- **自动获取现有维度**：脚本启动时自动从API获取所有现有维度
- **自动创建新维度**：如果输入文件中包含不存在的维度，脚本会自动创建
- **智能缓存机制**：维度映射缓存在内存中，提高处理效率

无需运行任何初始化命令，直接使用即可！

## 使用方法

### 1. 准备数据文件

创建制表符分隔的 `.txt` 文件，格式如下：

```
日期	开始时间	结束时间	分钟数	维度	活动描述
2025-08-08	12:20	12:25	5	5-财富	记录日志
2025-08-08	14:30	15:00	30	4-学习	阅读技术文档
```

### 2. 命令行使用

#### 基本用法（预览模式）

```bash
cd scripts
python batch_time_import.py sample_time_data.txt
```

#### 生成 JSON 输出文件

```bash
python batch_time_import.py sample_time_data.txt -o processed_data.json
```

#### 实际提交到 API

```bash
python batch_time_import.py sample_time_data.txt --submit
```

#### 指定 API 地址

```bash
python batch_time_import.py sample_time_data.txt --api-url http://your-api-server:8000 --submit
```

#### 启用详细日志

```bash
python batch_time_import.py sample_time_data.txt -v --submit
```


## 最佳实践

### 1. 数据准备

- 确保输入文件使用 UTF-8 编码
- 检查日期和时间格式是否正确
- 验证维度名称是否与系统匹配

### 2. 测试流程

1. 首先使用预览模式检查数据
2. 查看生成的 JSON 文件确认格式正确
3. 小批量测试实际提交
4. 确认无误后进行大批量导入

### 3. 监控和调试

- 查看日志文件了解处理进度
- 使用 `-v` 参数获取详细日志
- 检查 API 响应确认数据正确提交

## 批量笔记导入脚本（`bulk_import_user_notes.py`）

该脚本面向管理员，用来指定 **用户 UUID**，为其确保某个标签存在，并批量导入多条笔记（每条都会自动附加该标签）。

### 使用前准备

1. 在仓库根目录激活 backend 对应的虚拟环境，并确认已安装 `backend/pyproject.toml` 中列出的依赖（至少需要 `pydantic`、`SQLAlchemy`）。
2. `.env` 或环境变量中需包含数据库连接信息，例如 `DATABASE_URL`、`SCHEMA_NAME` 等；脚本会自动尝试读取仓库根目录与 `backend/.env`。
3. 数据文件可使用以下任一格式：
   - JSON 数组：`[{"content": "第一条"}, {"content": "第二条", "person_ids": ["..."]}]`
   - 带 `notes` 键的 JSON 对象：`{"notes": ["纯文本", {"content": "另一条", "task_id": "..."}]}`
   - JSON Lines：每行一个 JSON 对象
   - 纯文本：使用至少三个连续的空行分隔笔记；若未出现分隔符，则整份文本会被视为单条笔记
4. 如果无法直接拿到用户 UUID，可以只提供 `--user-email`，脚本会自动查询对应的用户并打印其 ID。

### 典型命令

```bash
python scripts/bulk_import_user_notes.py \
  --user-email admin@example.com \
  --tag-name OnboardingNotes \
  --input-file /tmp/new_notes.json \
  --tag-description "批量导入" \
  --dry-run
```

去掉 `--dry-run` 即可执行真实导入。常用可选项：

- `--continue-on-error`：笔记校验/写入失败时继续处理剩余记录
- `--max-count N`：仅处理前 N 条以便抽样验证
- `--encoding`：纯文本输入的文件编码
- `--verbose`：输出更详细的调试日志

### 输入示例

**JSON 数组**（一条纯文本 + 一条带额外字段）：

```json
[
  "第一条笔记",
  {
    "content": "第二条，关联已有任务",
    "task_id": "c4973091-2c34-4b66-8a2b-2f2b74652eed",
    "tag_ids": ["额外标签 UUID"]
  }
]
```

**纯文本**（至少三个连续空行分隔多段；没有分隔符则视为单条）：

```
第一条笔记内容

第二条笔记，会整段写入

第三条……
```

脚本会为每条记录生成 `NoteCreate` 请求，自动附加 `--tag-name` 对应的标签 ID，缺失时会先创建该标签（dry-run 模式下只打印提示，不落库；标签名称沿用管理员输入，不再额外改变大小写；纯文本模式下若希望拆分多条，请在相邻笔记之间插入 ≥3 个连续空行，否则整份文本将被视为一条笔记）。
